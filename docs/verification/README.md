# v1 manual verification (handoff spec §3.6 AC#9)

Evidence that a real run streams, renders, persists, and replays through the full
stack (gateway + `fake-tau-serve` mock + React UI).

- `trace-complete.png` — a completed `greeter` run: turn → `fs-read` tool node (ok),
  assistant text, token summary.
- Playwright `web/test-results/**/video.webm` + `trace.zip` (gitignored) — the live
  build of the trace, regenerated by `pnpm e2e`.

## Reproduce
1. `cargo build`
2. `cargo run -p tau-gateway -- --project ./fixtures/demo --tau-bin ./target/debug/fake-tau-serve --no-sandbox`
3. `cd web && pnpm dev`, open http://localhost:5173, pick `greeter`, enter a prompt, Run.

Or run the automated proof: `cd web && pnpm e2e` (boots gateway+mock+Vite, drives Chromium).

## AC#2 note
The spec's <250ms target is measured *after the engine emits*. The e2e `firstPaint`
budget is end-to-end (click → first assistant text) and includes the mock's per-step
sleeps (~40–80ms), so it is an approximation against the mock, not a true
engine→browser measurement. For that, instrument the WS `onmessage` timestamp against
the gateway's emit log when the real `tau serve` engine lands.

## Known v1 limitation surfaced here
Reopening a *finished* run replays the span tree + per-span I/O (inspector), but the
streamed assistant *prose* is live-only in v1 (events aren't in the replay snapshot).
See `docs/known-limitations-v1.md`.
